The Problem
No standard packaging for AI4EO datasets

Every EO benchmark defines its own folder layout, metadata format, and loading logic. PyTorch, Hugging Face, and Grain all assume samples are enumerable, indexable, and structurally uniform, but no format guarantees this. Researchers write custom data loaders for every dataset. At foundation-model scale, this becomes a hard ceiling.

STAC stores one JSON per sample. A million samples means a million files. STAC-GeoParquet fixes queries but flattens the hierarchy and does not bundle data. Neither provides structural validation or constant-time access to data.

The Solution
Declare a contract, build your dataset

TACO is a specification for packaging AI-ready EO datasets. Before writing any data, you declare a contract (σ, μ) that fixes the shape of the entire dataset. Every sample is validated against this contract on insertion.

Contract = (σ, μ)
σ = structure tree · μ = metadata schema

Structure (σ) defines the file tree inside each sample. Fixed leaves like B02.tif and variable leaves like img*[4,16].tif. All samples share the same tree.

Metadata (μ) assigns typed fields to every non-leaf node. Each level generates its own Parquet table, queryable with SQL. No server needed.

Tradeoffs

Structural immutability. The contract (σ, μ) is fixed at creation time. Adding new file types or metadata fields requires a new dataset version. STAC is maximally flexible: any sample can carry any assets and properties. TACO trades that flexibility for strong guarantees — constant-time access, schema validation, and reproducibility. If your workflow needs per-sample variability, use STAC. If you need datasets that behave like database tables, use TACO.

How It Works
Three technologies, one pipeline
📦
Cloud-Optimized ZIP

Index header at byte 0 with offsets to every Parquet file. One range request reads the header. No Central Directory scan.

🗂️
Apache Parquet

Metadata loaded and filtered with SQL. Predicate pushdown skips row groups. Rows contain byte offsets to data files.

🔗
VSI Paths

Offset + size build /vsisubfile/ paths. GDAL reads exact bytes. No extraction needed.

Physical Layout
dataset/ ├── COLLECTION.json ← contract (σ, μ) + dataset metadata ├── METADATA/ │ ├── collection.parquet ← 1 row per sample │ ├── sample.parquet ← 1 row per child per sample │ ├── sample__before.parquet │ └── sample__after.parquet └── DATA/ ├── 0/ │ ├── before/ │ │ ├── B02.tif │ │ └── B03.tif │ ├── after/ │ │ ├── B02.tif │ │ └── B03.tif │ └── change_map.tif ├── 1/ ... └── N/ ...

Parquet filenames mirror metadata keys with __ replacing /. ZIP and FOLDER containers are interchangeable. Same data, different packaging.

SQL Interface
INSTALL taco FROM community;
LOAD taco;

-- All samples, one row per sample, VSI paths as columns
SELECT * FROM read_taco('cloudsen12.zip');

-- Filter with SQL. Predicate pushdown to Parquet
SELECT * FROM read_taco('cloudsen12.zip')
 WHERE cloud_cover < 10 AND split = 'train';

-- Single sample or range
SELECT * FROM read_taco('dataset.zip', idx=5);
SELECT * FROM read_taco('dataset.zip', idx=[0,100]);
Design Principles

Self-contained. No server, no database, no external catalog. Contract-first. Structure declared before data is written. Cloud-optimized. Partial reads via byte-range access. Language-agnostic. DuckDB extension works from Python, R, Julia. FAIR-compliant. Persistent IDs, licensing, spatial and temporal extent.
